# Contributing to paper-xlsx

Contributions are welcome. This fork runs on a stricter discipline than most
libraries, because its entire value proposition is that a save can never be
quietly wrong. The rules below are the ones CI enforces and reviewers hold the
line on; reading them first will save you a review round-trip.

## Ground rules

1. **Strict superset.** Upstream openpyxl's own pytest suite stays green on
   every PR — that is the mechanical proof that every existing caller keeps
   working. Behavioral deltas to stock openpyxl are limited to the two
   sanctioned ones (preserve mode, and loud `LossySaveWarning`s on the stock
   save path). Don't propose a third without an issue discussion first.
2. **Fail loudly; silence is the enemy.** Every operation has three legal
   outcomes: done correctly, refused with a typed error saying what was found
   and why it was unsafe, or done with a loud warning enumerating exactly what
   could not be preserved. Nothing may silently drop, silently skip, or
   silently corrupt.
3. **Refusal atomicity.** Validate fully, then mutate. A refused operation
   leaves the in-memory model, the dirty ledger, and any file on disk exactly
   as they were. Every documented refusal condition gets a test asserting both
   that the typed refusal is raised and that output bytes equal input bytes.
4. **The name rule.** The PyPI distribution is `paper-xlsx`; the import name is
   `openpyxl`, frozen forever. Never rename the package directory; never write
   `import paper_xlsx` anywhere, including tests and docs.
5. **No formula calculation engine — not one function.** A partial engine is a
   silent-wrongness machine. Computation is delegated to the LibreOffice
   oracle (`openpyxl/oracle.py`), which reports measurements, never judgments.

## Getting set up

```bash
git clone https://github.com/paper-instruments/paper-xlsx.git
cd paper-xlsx
pip install -e .
pip install pytest "lxml>=6.1,<7" pillow pandas
pytest -q
```

The LibreOffice-dependent tests (`lo_smoke` and the oracle suite) skip
automatically when `soffice` is not on your PATH; install `libreoffice-calc`
to run them locally. CI runs the full matrix: Python 3.9–3.13, a LibreOffice
leg, a default-dependency leg (no lxml), a stdlib-XML leg, Windows, a Sphinx
docs build, and a build/install/twine-check leg — see
[`.github/workflows/test.yml`](.github/workflows/test.yml).

## Testing discipline

- **Save → reopen → assert.** Every assertion about workbook content runs
  against a saved-and-reopened file, never the in-memory object you just
  mutated.
- **The contract harness's five assertions.** Fork behavior tests assert:
  (1) save → reopen → assert, never in memory; (2) the intended effect is
  present in the reopened workbook; (3) the **changed-part budget** — a package
  diff shows exactly the expected parts changed, per the operation's sanctioned
  collateral set, and every other part payload byte-identical; (4) an
  independent-loader smoke — headless LibreOffice load/convert where `soffice`
  exists; (5) refusal atomicity — every documented refusal input raises the
  typed refusal and leaves output bytes identical to input bytes.
- **No fix without a fixture.** Every bug found downstream becomes a frozen
  fixture plus a failing test *before* its fix merges.
- **Golden files** update only via explicit command with human-reviewed diffs;
  anything date-stamped takes an injectable clock.

## The fixture corpus

Fixtures live in `tests/paper/fixtures/`, frozen and hash-pinned via
`MANIFEST.sha256`, each with a JSON sidecar recording exact provenance
(authoring application and version, features present, known ground-truth
values, human verifier and date). Rules:

- Fixture bytes are **never generated by code under test** and never
  regenerated casually.
- **Never label a fixture with provenance it lacks.** A LibreOffice-authored
  file is labeled LibreOffice-authored, not "Excel."
- Adding a fixture means adding its sidecar and updating `MANIFEST.sha256` in
  the same commit.

The most valuable contribution many people can make is not code: it is
**real application-authored workbooks** — files from desktop Excel or Google
Sheets that our environment cannot author. The current wish list, and the
acceptance procedure, are in [FIXTURE-REQUESTS.md](FIXTURE-REQUESTS.md).

## House style

- New XML vocabulary uses upstream's own declarative mapping framework — the
  `Serialisable`/descriptor system in `openpyxl/descriptors/` — never
  hand-assembled element trees in API code, and **never string-formatted XML
  anywhere**. The splice writer emits through the same event-writing machinery
  upstream uses (et_xmlfile/lxml).
- Match upstream's code style in upstream files; the fork's own modules live
  under `openpyxl/preserve/`, `openpyxl/package/`, `openpyxl/errors.py`, and
  `openpyxl/oracle.py`.
- Programmer errors stay `TypeError`/`ValueError`; environmental and safety
  refusals are `PaperRefusal` subclasses from `openpyxl/errors.py`.

## Submitting changes

1. Open an issue first for anything beyond a small fix, especially anything
   touching the save path, the ledger, or the refusal taxonomy.
2. Branch from `main`, keep PRs focused, and include the fixture + failing
   test with every bug fix.
3. Make sure `pytest -q` passes locally, including the upstream suite.
4. Note in the PR description which operation classes your change touches and
   what the expected changed-part budget is.

Docs for the preserve-mode surface live in [`doc/paper.rst`](doc/paper.rst);
user-visible changes need a matching entry in [`doc/changes.rst`](doc/changes.rst)
under "Unreleased".
